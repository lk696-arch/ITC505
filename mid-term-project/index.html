Planning & Structure

I designed the story as a directed graph of stages, where each stage has text, an image, and a set of choices that connect to the next stage via a key. The script.js file keeps this data in a single JavaScript object named story. A render() function updates the DOM with the current stage, generating buttons with event listeners for each choice.

Debugging & Testing

I tested every branch to ensure all endings were reachable and each endpoint displayed the correct image. I added basic input guards and simple fallbacks for missing images. The UI uses clear hierarchy and large tap targets to minimize errors on mobile devices.

Design Decisions

  • Colorful, simple theme: A friendly palette, soft card shadows, and plenty of whitespace for readability.
  • Accessibility: Semantic landmarks, live region updates, and ALT text on images.
  • Maintainability: Data‚Äëdriven scenes allow new chapters/endings to be added by editing one object.

Challenges & Resolutions

  • Branch complexity: I mapped branches first, then encoded them as keys to avoid dead links.

Insights Gained

Separating content (story data) from logic (render/update functions) makes iterative development far easier. DOM manipulation with event listeners is a clean fit for branching narratives.
